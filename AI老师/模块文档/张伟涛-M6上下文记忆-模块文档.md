## 一、文档概述

### 1.1 编写目的

本文档用于明确 LeapMind 教育网站 M6“上下文记忆（全局）”模块的建设目标、业务范围、功能需求、数据结构、接口契约、非功能要求和验收标准，为产品设计、前后端开发、算法实现、测试和项目验收提供统一依据。

M6 不直接完成做题、讲题或问答业务，而是持续接收用户在各学习模块中的行为数据，将分散的学习记录转化为可查询、可更新、可解释的学习画像，并在获得授权的情况下向其他模块提供必要的个性化上下文。

### 1.2 项目背景

传统教育类 AI 往往只理解当前问题，无法稳定记住学生长期的学习情况。例如，同一名学生多次在“勾股定理”中混淆斜边与直角边，系统若只处理单次对话，后续仍可能重复给出不适合该学生的标准答案。

M6 通过统一采集答题、讲题反馈、讲课互动和对话等行为，形成学生的知识掌握状态、近期困惑、讲解偏好、学习节奏和复习计划。其他 AI 模块调用大模型前，可获取经过压缩和脱敏的画像摘要，使讲解内容更加连贯和个性化。

### 1.3 术语说明

| 术语 | 说明 |
|------|------|
| 学习事件 | 用户在各模块产生的一次可记录行为，如答题、追问、完成讲课或提交反馈 |
| 短期记忆 | 当前会话或近期学习任务所需的上下文，主要服务于连续交互 |
| 长期记忆 | 跨会话保存的稳定学习信息，如知识掌握度、长期困惑和偏好趋势 |
| 用户画像 | 根据多源学习事件计算得到的结构化学习特征集合 |
| 画像摘要 | 从完整画像中提取的精简文本或结构化数据，用于注入 AI Prompt |
| 掌握度 | 用户对某知识点的综合掌握水平，取值范围为 0～1 |
| 记忆置信度 | 系统对某项推断结果的可信程度，取值范围为 0～1 |
| 复习计划 | 根据学习和复习记录计算的知识点复习阶段与下次复习时间 |

## 二、建设目标与范围

### 2.1 建设目标

1. 建立统一的学习行为事件采集机制，打通 M1、M2、M3、M4、M5 和 M7 的数据。
2. 建立可持续更新的用户学习画像，识别擅长点、薄弱点、近期困惑、学习节奏和讲解偏好。
3. 为 M2、M4、M5 和 M7 提供统一、轻量的画像查询与 Prompt 上下文注入能力。
4. 根据知识点掌握和复习行为生成复习计划，降低知识遗忘带来的学习损失。
5. 保障用户对长期记忆的知情权、查看权、更正权、删除权和关闭权。
6. 确保画像结论具备数据来源、更新时间和置信度，避免将临时行为误判为长期特征。

### 2.2 模块范围

本期范围包括：

- 多模块学习事件的单条与批量采集；
- 学习事件校验、去重、持久化和异常记录；
- 知识点掌握度、优势、薄弱点、近期困惑、学习节奏与讲解偏好的计算；
- 完整画像、画像摘要和知识点掌握状态查询；
- 画像在 AI 请求中的按需注入；
- 复习计划生成、到期提醒和完成反馈；
- 学习档案、记忆设置和复习提醒的前端展示；
- 画像数据的查看、更正、删除、停用和审计。

### 2.3 非本期范围

- 医学、心理学或智力水平诊断；
- 仅凭少量行为对用户进行固定人格分类；
- 向其他用户公开个人画像或进行学生间负面比较；
- 自动替代教师作出升学、分班、奖惩等高影响决策；
- 复杂深度学习推荐模型及跨学校画像共享；
- 未经用户或监护人授权采集与学习无关的设备、位置和社交数据。

### 2.4 用户角色

| 角色 | 主要诉求 | 权限范围 |
|------|----------|----------|
| 学生 | 获得连续、个性化的学习支持 | 查看本人画像、复习计划和记忆设置；更正或删除本人记忆 |
| 教师 | 了解学生学习状态，调整教学内容 | 在授权范围内查看所负责学生的教学相关画像，不查看无关对话原文 |
| 监护人 | 了解未成年学生的学习趋势 | 经授权查看汇总趋势和复习提醒，不查看不必要的私密对话内容 |
| 管理员 | 维护系统配置和处理数据请求 | 管理事件类型、保留策略和权限；操作必须记录审计日志 |
| 其他业务模块 | 获取完成业务所需的最小画像上下文 | 通过服务接口按场景查询，不得直接读取全部底层事件 |

## 三、总体业务设计

### 3.1 记忆分层

M6 将上下文记忆分为三层：

| 层级 | 保存内容 | 生命周期 | 主要用途 |
|------|----------|----------|----------|
| 会话记忆 | 当前会话问题、引用对象、最近消息摘要 | 会话结束后按策略过期 | 保证连续对话和指代消解 |
| 阶段记忆 | 最近 7～30 天关注点、近期困惑和学习任务 | 随新事件滚动更新 | 支持近期学习计划和针对性讲解 |
| 长期画像 | 知识掌握趋势、稳定偏好和复习状态 | 用户启用期间持续维护 | 支持跨模块个性化和长期学习跟踪 |

临时情绪、偶发错误和单次表达不得直接写成稳定画像。系统只有在样本数量、时间跨度和置信度达到配置阈值后，才能形成长期结论。

### 3.2 模块关系

```text
M1 答题记录 ──────┐
M2 讲题反馈 ──────┤
M3 薄弱点结果 ────┤
M4 讲课互动 ──────┼──> M6 事件采集 ──> 画像计算 ──> 用户画像与复习计划
M5 备课使用反馈 ──┤                              │
M7 对话与追问 ────┘                              └──> M2/M4/M5/M7 AI 上下文
```

### 3.3 核心业务流程

1. 用户在某业务模块中完成答题、提问、反馈或复习等操作。
2. 来源模块生成唯一事件 ID，并将标准化事件提交给 M6。
3. Java 事件采集服务完成身份校验、字段校验、幂等判断和数据持久化。
4. 关键事件进入增量计算队列；非关键事件等待定时批处理。
5. Python 画像引擎聚合近期数据，更新知识掌握度、困惑点、偏好和复习状态。
6. 画像结果写入 MySQL，并删除 Redis 中对应用户的旧缓存。
7. 其他模块发起 AI 请求时，由 Java 服务根据场景获取最小画像摘要。
8. Python AI 服务将画像摘要作为辅助上下文构建 Prompt，并明确其优先级低于用户当前指令。
9. 前端学习档案页展示画像依据、更新时间和可执行的复习建议。

### 3.4 画像更新策略

- 答错、连续追问、“仍然不懂”等关键事件在 5 分钟内触发增量更新。
- 普通浏览、播放和停留事件按小时聚合，避免频繁计算。
- 每天凌晨执行一次全量校准，修正增量计算可能产生的偏差。
- 每项推断结果必须保存样本数、计算窗口、更新时间和置信度。
- 新用户或数据不足时返回“证据不足”，不得生成确定性标签。
- 用户主动更正的偏好优先于算法推断，但应允许用户恢复为自动判断。

## 四、功能需求

### 4.1 FR-01 学习事件采集

#### 4.1.1 功能说明

系统应向各业务模块提供统一事件采集接口，支持实时单条上报和离线批量补报。

#### 4.1.2 事件公共字段

| 字段 | 必填 | 说明 |
|------|------|------|
| eventId | 是 | 来源模块生成的全局唯一事件 ID，用于幂等去重 |
| userId | 是 | 当前用户 ID，必须与登录身份或服务授权一致 |
| eventType | 是 | 标准事件类型 |
| sourceModule | 是 | M1、M2、M3、M4、M5 或 M7 |
| occurredAt | 是 | 事件实际发生时间，ISO 8601 格式 |
| sessionId | 否 | 所属学习会话 ID |
| kpId | 否 | 关联知识点 ID；无知识点场景可为空 |
| data | 是 | 不同事件类型对应的扩展字段 |
| schemaVersion | 是 | 事件结构版本，初始值为 `1.0` |

#### 4.1.3 核心事件类型

| 事件类型 | 来源 | 关键数据 |
|----------|------|----------|
| answer_question | M1 | 是否正确、耗时、尝试次数、答案类型 |
| finish_practice | M1 | 题目数、正确率、总耗时 |
| request_explanation | M2 | 知识点、问题来源、讲解方式 |
| explanation_feedback | M2 | 是否理解、反馈标签、再次讲解次数 |
| weak_point_changed | M3 | 原掌握度、新掌握度、变化原因 |
| lecture_interact | M4 | 暂停、回看、提问、章节位置 |
| lesson_material_used | M5 | 使用的材料类型和反馈结果 |
| ask_doubt | M7 | 知识点、困惑标签、是否为追问 |
| mark_reviewed | M6 | 复习知识点、完成结果、复习耗时 |
| preference_changed | M6 | 用户主动设置的讲解偏好 |

#### 4.1.4 业务规则

- 相同 `eventId` 重复提交时只处理一次，并返回原处理结果。
- 路径中的 `userId` 与请求体、登录身份不一致时拒绝请求。
- 不保存完整访问令牌、密码、身份证号等敏感信息。
- 对话事件默认只保存知识点、意图、困惑标签等结构化结果；确需保存原文时必须单独授权并设置保留期限。
- 批量接口单次最多接收 100 条事件，部分失败时返回每条事件的处理结果。
- 未知 `eventType` 或不支持的 `schemaVersion` 应返回明确错误，不得静默丢弃。

### 4.2 FR-02 用户画像计算

#### 4.2.1 知识掌握度

系统应根据正确率、答题次数、题目难度、提示使用情况、用时、近期趋势和复习结果计算知识点掌握度。建议初期采用可解释的加权模型：

```text
masteryScore = 0.35 × accuracyScore
             + 0.20 × difficultyScore
             + 0.15 × stabilityScore
             + 0.15 × independenceScore
             + 0.15 × recencyScore
```

各子项归一化到 0～1。权重必须配置化，算法版本写入计算结果，不得硬编码在多个服务中。

掌握状态划分如下：

| 掌握度 | 状态 |
|--------|------|
| 0～0.39 | 薄弱 |
| 0.40～0.59 | 待巩固 |
| 0.60～0.79 | 基本掌握 |
| 0.80～1.00 | 熟练 |

当有效答题数少于 5 次时，系统应显示“数据不足”，不将知识点标记为擅长或薄弱。

#### 4.2.2 困惑点识别

系统应结合以下信号识别近期困惑：

- 同一知识点连续答错；
- 多次请求讲解或重复追问；
- 反馈“没听懂”“概念不清”“步骤不明”；
- 讲课中频繁暂停、回看或跳回同一内容；
- M3 已确认的薄弱点变化。

每个困惑点应包含知识点、具体描述、出现频次、最近出现时间、证据来源和置信度。困惑点默认按最近 30 天计算，并对旧数据执行时间衰减。

#### 4.2.3 学习偏好推断

系统可推断用户更常采用的内容形式和讲解方式，包括图示、语音、文字、例题演示、分步讲解和先结论后解释等。学习偏好属于动态辅助信息，不得解释为固定的“学习类型”。

- 算法推断结果必须展示置信度；
- 用户可以主动选择或修改偏好；
- 用户主动设置优先于算法推断；
- 数据不足时使用系统默认讲解方式；
- 每 30 天重新评估一次稳定偏好。

#### 4.2.4 学习节奏

系统应按同年级、同难度题目的合理用时区间，结合用户自身历史中位数推断学习节奏。结果可为 `slow`、`moderate`、`fast`，但对外展示应使用“偏细致”“适中”“偏快速”等中性表述。

#### 4.2.5 画像摘要

系统应同时生成结构化摘要和自然语言摘要。摘要只包含当前业务场景必需的信息，示例如下：

```json
{
  "userId": 1001,
  "grade": "grade_8",
  "weaknesses": [
    {
      "kpId": 10,
      "kpName": "勾股定理",
      "masteryLevel": 0.35,
      "confidence": 0.88
    }
  ],
  "recentConfusions": ["分不清斜边和直角边"],
  "preferredExplanationStyle": "step_by_step_with_diagram",
  "learningPace": "moderate",
  "lastUpdated": "2026-07-16T02:00:00+08:00"
}
```

自然语言摘要必须以事实和趋势为主，不得包含贬低性评价，例如将“近期错误较多”描述成“能力差”或“不认真”。

### 4.3 FR-03 画像查询

系统应提供以下查询能力：

- 查询用户完整学习画像；
- 查询适合 AI 注入的轻量画像摘要；
- 查询全部知识点掌握状态；
- 按学科、时间范围和状态筛选知识点；
- 查询画像项的数据依据、更新时间和置信度；
- 数据不足时返回明确状态，而不是空白或虚构内容。

完整画像仅用于学习档案页和授权教师端；业务模块默认只能获取按场景裁剪后的摘要。

### 4.4 FR-04 AI 上下文注入

#### 4.4.1 注入规则

- M2 讲题：注入当前知识点掌握度、最近错误原因和讲解偏好。
- M4 讲课：注入课程相关薄弱点、近期关注点和内容形式偏好。
- M5 备课：仅在教师拥有对应学生或班级权限时注入聚合画像，不默认注入单个学生私密数据。
- M7 问答：注入当前会话摘要、相关知识点困惑和用户主动设置的讲解偏好。
- 与当前问题无关的画像项不得注入。
- 用户当前明确要求与历史偏好冲突时，以当前要求为准。

#### 4.4.2 Prompt 安全规则

- 画像内容以结构化数据区块传入，不与系统指令混写。
- 用户历史文本按不可信输入处理，防止其中内容被解释为系统指令。
- 注入前进行长度控制、敏感字段过滤和场景权限校验。
- 每次注入记录调用模块、画像版本、字段范围和时间，但日志中不得记录不必要的隐私原文。
- 画像服务不可用时，AI 调用应降级为无画像模式，不得阻断用户正常学习。

### 4.5 FR-05 复习计划与提醒

#### 4.5.1 复习计划

系统应在用户首次学习、掌握度下降或完成复习后更新知识点复习计划。MVP 默认采用 1 天、3 天、7 天、14 天和 30 天的阶段间隔，并根据复习结果动态调整：

- 复习正确且能独立完成：进入下一阶段；
- 复习正确但依赖提示：保持当前阶段；
- 复习错误或仍不理解：回退至第一阶段；
- 用户主动推迟：记录推迟原因并重新安排，不判定为学习失败。

#### 4.5.2 提醒展示

- 学习档案页展示今日待复习、即将到期和已逾期项目；
- 用户可按学科筛选，并跳转至 M1 练习或 M2 讲题；
- 同一知识点同一天最多产生一次站内提醒；
- 用户可关闭提醒或设置免打扰时间；
- 本期默认采用站内提醒，短信、邮件和第三方推送不在范围内。

### 4.6 FR-06 学习档案前端

学习档案页应包含：

1. 画像摘要：展示近期学习状态、优势、待巩固方向和更新时间。
2. 知识掌握视图：按学科展示知识树、掌握度和置信度。
3. 近期困惑：展示困惑描述、相关知识点和推荐操作。
4. 学习趋势：展示一段时间内掌握度、练习量和复习完成率变化。
5. 复习提醒：展示待复习知识点和跳转入口。
6. 画像依据：允许用户查看某项结论由哪些类型的数据得出。
7. 记忆设置：管理长期记忆、对话原文使用、个性化注入和提醒开关。

前端不得只使用颜色表示状态，所有图表必须同时提供文字或图标说明。

### 4.7 FR-07 用户记忆控制

用户应具备以下控制能力：

- 查看系统保存的主要长期记忆及其用途；
- 更正讲解偏好、年级等可编辑信息；
- 删除单条困惑记忆或清空全部长期记忆；
- 关闭新的长期记忆采集；
- 单独关闭画像注入，但保留基础学习记录；
- 申请导出本人画像和复习计划；
- 查看删除任务的处理状态。

关闭长期记忆后，系统不再产生新的长期画像，但完成做题等核心业务所需的短期数据可按最小原则继续处理。删除请求应在 24 小时内完成在线数据删除，备份数据按备份周期自然过期。

### 4.8 FR-08 权限与审计

- 学生只能访问本人画像。
- 教师只能访问其负责班级或已授权学生的教学相关画像。
- 监护人访问未成年人画像必须具有有效绑定关系和授权记录。
- 管理员不得通过普通业务接口查看用户画像内容。
- 查询、导出、更正、删除和管理员配置操作必须记录审计日志。
- 权限撤销后，后续请求立即失效，缓存中的授权结果同步清除。

### 4.9 FR-09 异常与降级处理

| 异常场景 | 系统处理 |
|----------|----------|
| 重复事件 | 返回已接收结果，不重复计算 |
| 非法字段或未知事件类型 | 返回参数错误及具体字段位置 |
| Python 画像引擎不可用 | 保留事件并进入重试队列，继续提供上一版本画像 |
| Redis 不可用 | 回源 MySQL 查询并记录告警 |
| 画像数据不足 | 返回 `insufficient_data`，使用默认教学策略 |
| 画像计算超时 | 保留旧版本并标记更新时间，不覆盖有效数据 |
| 用户已关闭个性化 | AI 请求不注入画像，仅使用当前问题上下文 |
| 删除任务失败 | 自动重试并向用户展示处理状态，不返回虚假成功 |

## 七、技术方案

### 7.1 技术分工

| 层级 | 技术 | 主要职责 |
|------|------|----------|
| 前端 | React + TypeScript | 学习档案、知识掌握视图、复习提醒和记忆设置 |
| Java 后端 | Spring Boot、MyBatis-Plus、Redis | 鉴权、事件采集、查询接口、缓存、审计和任务编排 |
| Python 服务 | FastAPI、pandas、numpy、scikit-learn | 多源聚合、画像计算、困惑提取和摘要生成 |
| 数据层 | MySQL + Redis | 事件、画像、掌握度、复习计划持久化与热点缓存 |
| AI 能力 | 统一大模型调用层 | 画像摘要和学习建议生成，不直接承担关键数值计算 |

### 7.2 设计原则

- Java 负责可靠的业务入口和权限控制，Python 负责画像算法与自然语言处理。
- 数值型掌握度由可测试、可版本化的算法计算，不直接让大模型自由判断。
- 大模型生成的摘要必须基于结构化事实，并通过长度、敏感词和主观评价检查。
- 事件原始记录与画像计算结果分离，便于算法升级后重新计算。
- 所有画像结果均携带版本，支持灰度升级和问题回滚。

### 7.3 任务调度

- 增量任务：关键事件进入消息队列或任务表，目标处理延迟小于 5 分钟；
- 小时任务：聚合播放、停留等高频低价值事件；
- 日任务：全量校准用户画像和复习计划；
- 清理任务：按数据保留策略清理过期事件、摘要缓存和删除请求关联数据；
- 失败任务：指数退避重试 3 次，仍失败后进入死信队列并告警。

## 八、非功能需求

### 8.1 性能

| 指标 | 要求 |
|------|------|
| 单条事件采集接口 | P95 响应时间不超过 300 ms |
| 批量事件采集接口 | 100 条事件 P95 不超过 1 s |
| 画像摘要查询 | Redis 命中时 P95 不超过 100 ms；回源时不超过 500 ms |
| 完整画像查询 | P95 不超过 800 ms |
| 关键事件画像更新 | 95% 在事件发生后 5 分钟内完成 |
| 系统可用性 | 月度可用性不低于 99.9% |

### 8.2 数据一致性

- 事件采集采用至少一次提交和服务端幂等处理。
- 画像写入采用版本号控制，旧任务不得覆盖新画像。
- MySQL 更新成功后再清理缓存，缓存异常不得影响数据库结果。
- 复习标记与复习计划更新应处于同一事务或具备可靠补偿机制。

### 8.3 安全与隐私

- 全部外部接口使用 HTTPS，内部服务使用受控内网和服务认证。
- 数据库中的敏感字段采用加密或脱敏存储，日志禁止输出访问令牌和完整隐私文本。
- 遵循数据最小化原则，只采集完成学习服务所必需的数据。
- 未成年人场景应支持监护人授权和适龄化隐私说明。
- 画像不得作为惩罚、歧视或自动化高影响决策的唯一依据。
- 用户关闭或撤回授权后，系统必须停止对应范围的新数据处理。

### 8.4 可解释性

- 每项画像结论应能追溯到事件类型和统计区间。
- 前端应展示“系统为什么得出该结论”的简要说明。
- 所有算法结果保存算法版本、样本量、置信度和计算时间。
- 用户更正画像后，应记录更正来源，避免下次计算立即覆盖。

### 8.5 可维护性

- 事件类型、权重、阈值、复习间隔和数据保留期限统一配置化。
- Java 与 Python 使用版本化接口契约，并提供契约测试。
- 核心函数和公共接口应具有中文 Javadoc 或 Docstring。
- 生产环境日志包含时间、模块、级别、请求 ID 和必要错误堆栈，不输出敏感信息。

### 8.6 可访问性

- 页面支持键盘操作和清晰的焦点状态。
- 图表提供文字替代说明，不仅依赖颜色区分。
- 重要提示满足合理的颜色对比度，并适配移动端页面。

## 十一、潜在风险与应对措施

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 数据量少导致误判 | 画像不准确，影响讲解体验 | 设置最小样本量和置信度，数据不足时使用默认策略 |
| 历史画像固化用户 | 系统忽略用户近期进步 | 使用时间衰减、趋势指标和定期重算，允许用户更正 |
| 多模块事件口径不一致 | 画像计算错误 | 建立统一事件字典、版本管理和契约测试 |
| AI 摘要产生主观评价 | 对用户造成误导或负面影响 | 摘要仅基于结构化事实，执行中性表达检查和人工抽检 |
| 隐私数据泄露 | 造成合规和信任风险 | 最小化采集、权限隔离、加密、脱敏、审计和删除机制 |
| 画像服务故障影响主业务 | AI 功能不可用 | 缓存上一版本画像，自动降级为无画像模式 |
| 高频事件导致计算压力 | 延迟升高、成本增加 | 事件分级、批量聚合、异步队列和热点缓存 |
| 复习提醒过多 | 用户产生打扰感 | 每日去重、免打扰时间、提醒开关和优先级控制 |

## 十三、结论

M6“上下文记忆（全局）”是 LeapMind 从单次 AI 工具升级为持续学习伙伴的关键基础模块。其核心不是无限保存用户信息，而是在明确授权、数据最小化和用户可控的前提下，将必要的学习行为转化为准确、可解释、可更新的学习上下文。

本报告采用“事件采集—画像计算—场景查询—AI 注入—反馈更新”的闭环设计，并通过短期、阶段和长期三层记忆控制数据用途。第一阶段优先交付可解释的规则模型、场景化画像摘要和复习提醒，后续再依据真实数据逐步引入复杂算法，从而兼顾开发可行性、教学价值、系统稳定性和隐私安全。

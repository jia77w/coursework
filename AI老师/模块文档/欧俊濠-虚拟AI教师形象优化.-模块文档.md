## 版本整合说明

本最终方案由 M8 V1.0 与 V2.0 合并、去重和统一后形成。整合原则如下：

1. **保留 V1.0 的基础交付要求**：Live2D 模型渲染、基础表情与动作、TTS 语音合成、音频缓存、口型同步、待机/讲课/鼓励/思考状态、至少两种教师形象与音色、普通设备 30fps 目标，以及与 M2、M4、M7 的联动。
2. **吸收 V2.0 的体验与技术优化**：面向大学生的教师人格、微动作与倾听感、低频趣味互动、小白板与课程道具、人物空间移动、教学锚点、遮挡检测、视线与朝向控制、任务取消机制和分级性能降级。
3. **统一目标用户**：当前版本只面向大学生学习用户，不再设计小学、初中和高中差异化教师风格。
4. **控制实施范围**：最终方案不采用复杂三维数字人或游戏级自由寻路，而采用“预设站位＋教学锚点＋候选位置评分＋Live2D 内部动作＋模型外部坐标移动”的可落地方案。
5. **分阶段交付**：基础能力作为必须完成项，空间移动与自然交互作为重点创新项，高精度音素口型、复杂道具和完整全身行走作为后续优化项。

## 一、文档概述

### 1.1 编写目的

本文档用于明确 LeapMind 教育网站 M8“虚拟 AI 教师形象（优化）”模块的建设目标、业务范围、交互设计、功能需求、状态模型、技术方案、接口契约、数据结构、非功能要求、验收标准和实施计划，为产品设计、前端开发、后端开发、Live2D 模型制作、AI 服务接入、测试和项目验收提供统一依据。

M8 不直接负责生成课程内容、解答题目或保存学生画像，而是接收 M2 AI 讲题、M4 即时讲课、M6 上下文记忆和 M7 实时问答等模块提供的教学内容、学生状态与业务事件，将其转换为虚拟教师的语音、口型、表情、动作、视线、站位和空间移动表现。

本最终整合版在 V1.0 基础能力之上，纳入并统一以下 V2.0 优化：

1. 将目标用户统一调整为大学生，不再区分小学、初中和高中风格；
2. 增加大学生适用的教师人格和表达风格；
3. 增加微动作、视线、倾听、停顿和连续交流等“人感”设计；
4. 增加轻量幽默、小白板、课程道具和互动挑战等趣味功能；
5. 突破固定侧边栏限制，使人物能够在教学页面中移动；
6. 建立教学舞台、预设站位、教学锚点、遮挡检测和空间行为控制；
7. 明确 Live2D、Canvas/WebGL、状态机、DOM 锚点、Web Audio API 等技术实现方式；
8. 建立完整模式、轻量模式、固定模式和静态模式等性能降级机制。

### 1.2 项目背景

多数 AI 教育产品主要通过文本框、语音播放器或固定头像与学生交互。即使 AI 已经能够完成讲课、讲题和答疑，学生仍容易感到它只是一个功能工具，而不是正在参与教学过程的教师。

传统固定虚拟形象还存在以下问题：

- 人物只在角落中循环播放动作，与讲解内容没有关系；
- 表情和动作切换机械，缺乏倾听、思考、停顿等真实交流节奏；
- 语音音色单一，句子没有重点、重读和合理停顿；
- 人物不能靠近公式、图表、题目和代码，无法参与页面讲解；
- 动作过多会干扰阅读，动作过少又会显得僵硬；
- 低配置设备容易出现模型加载慢、帧率低和音画不同步；
- 面向大学生时，过度卖萌和夸张表扬会降低专业感。

M8 最终方案将虚拟教师从“固定动态头像”升级为“教学场景中的数字教师”。系统通过统一的教师表现引擎，根据教学语义、学生行为、页面布局和设备性能，控制人物在合适的位置，以合适的语气、表情、动作和节奏参与学习。

### 1.3 产品定位

M8 面向大学生学习场景，虚拟教师应满足以下定位：

> 专业但不严肃，亲切但不幼稚；能够自然讲解、认真倾听、适度鼓励，并通过克制的幽默和空间互动提升学习兴趣。

虚拟教师不应被设计为单纯的二次元展示角色，也不应以频繁动作吸引注意力。人物的一切表现应优先服务于知识理解、注意力引导和学习陪伴。

| 术语 | 说明 |

| 降级模式 | 设备性能或服务异常时，逐步减少物理效果、移动和动画复杂度的运行模式 |

## 二、建设目标与范围

### 2.1 建设目标

1. 接入 Live2D 虚拟教师形象，实现稳定的模型加载、渲染和资源释放。
2. 建立统一状态机，支持初始化、待机、倾听、思考、移动、讲课、强调、鼓励、引导、暂停、完成和异常等状态。
3. 建立教师表现引擎，根据讲解内容和业务事件自动选择语音节奏、表情、动作、视线、停顿和站位。
4. 增加自然微动作和倾听反馈，减少固定循环动画带来的机械感。
5. 提供适合大学生的三种教师风格，避免低龄化表达和过度卖萌。
6. 提供活跃、标准和专注三档表现强度，适配互动学习、普通课堂和深度学习场景。
7. 接入 TTS 语音合成，支持至少两种音色、语速和音量设置。
8. 实现语音缓存、播放、暂停、继续、停止和用户打断。
9. 实现基于音频振幅的基础口型同步，并预留音素时间轴升级能力。
10. 建立虚拟教学舞台，使人物能够在预设站位和教学锚点之间自然移动。
11. 支持人物靠近公式、图表、代码和题目，并通过视线与手势参与讲解。
12. 建立遮挡检测和备用站位机制，避免人物覆盖主要学习内容。
13. 打通 M2、M4、M6 和 M7，形成讲题、讲课、记忆和实时问答联动。
14. 在普通设备上稳定达到平均 30fps 以上，并支持自动性能降级。
15. 至少提供两种可选教师形象和两种可选音色。
16. 在模型、语音、音频和浏览器能力异常时，保证核心学习功能仍可使用。

### 2.2 模块范围

本期范围包括：

- Live2D 模型加载、渲染、缩放、朝向和位置控制；
- 模型表情、动作和参数映射；
- 呼吸、眨眼、视线和身体重心等微动作；
- 教师状态机和动作优先级；
- 教师表现脚本和教学语义标签；
- 大学生教师风格与表现强度配置；
- TTS 语音生成、分段、播放、缓存和失败重试；
- 音量驱动口型同步；
- 教学舞台、预设站位和教学锚点；
- 人物转身、移动、到达、指向和返回流程；
- 候选站位评分和基础遮挡检测；
- 用户拖动、锁定、缩放、隐藏和恢复位置；
- 小白板、公式卡、代码提示和成就卡等轻量道具；
- 低频幽默、互动提问和学习阶段反馈；
- M2、M4、M6、M7 联动；
- 性能监控、资源释放和自动降级；
- 模型、音色、动作、表情和用户设置的管理。

### 2.3 非本期范围

本期暂不包括：

- 三维数字人建模和三维空间导航；
- 摄像头驱动的实时动作捕捉；
- 面部识别、情绪识别或心理状态判断；
- 复杂游戏级 A* 自由寻路和动态碰撞物理；
- 多名虚拟教师同时讲课；
- 学生自行上传未经审核的模型；
- 在线自动完成高质量分层原画和 Live2D 绑定；
- 高精度全身三维转身和复杂舞蹈动作；
- 无人工审核的自由幽默生成；
- 让人物动作替代字幕、文字或主要知识内容；
- 使用学生画像作出成绩、处分或其他高影响决策。

|---|---|---|

| M6 上下文记忆模块 | 提供学生偏好和近期学习上下文 | 提供经过裁剪的讲解偏好、学习节奏和历史困惑摘要 |
| M7 实时问答模块 | 控制倾听、思考、回答和打断 | 发送实时对话状态、文本分段和打断事件 |

## 三、总体业务设计

### 3.1 总体设计原则

1. **教学优先**：动作和移动必须帮助理解，不以展示动画为主要目的。
2. **大学生定位**：语言自然、平等、克制，避免幼稚化和夸张表扬。
3. **状态统一**：业务模块不得直接随意操作模型参数，统一通过状态机和表现引擎控制。
4. **空间受控**：第一阶段采用预设站位和锚点附近移动，不做完全自由寻路。
5. **内容不遮挡**：人物始终避让主要知识内容和操作区域。
6. **用户可控**：学生可关闭语音、动作、自主移动、幽默和整个教师形象。
7. **逐级降级**：虚拟教师异常不得阻断文字、题目、课件和问答功能。
8. **配置驱动**：模型差异、动作名称和参数映射由配置管理，避免业务代码硬编码。

### 3.2 教师状态模型

| 状态 | 状态编码 | 主要表现 |
|---|---|---|
| 初始化 | initializing | 显示加载提示，加载模型、动作和用户设置 |
| 入场 | entering | 从侧面或默认区域进入教学舞台 |
| 待机 | idle | 呼吸、眨眼、轻微视线和重心变化 |
| 倾听 | listening | 停止语音，面向学生，轻微点头 |
| 思考 | thinking | 视线短暂移动、轻微思考动作，限制最大时长 |
| 转身 | turning | 看向移动方向并切换身体朝向 |
| 移动 | moving | 播放行走或过渡动作，并改变模型坐标 |
| 到达 | arrived | 减速、站稳、调整视线和朝向 |
| 讲课 | teaching | 播放语音、口型和普通讲解动作 |
| 强调 | emphasizing | 认真表情、降低语速、指向重点 |
| 提问 | asking | 疑问表情、停止讲解并等待学生回答 |
| 鼓励 | encouraging | 微笑、点头或轻度认可动作 |
| 引导 | guiding | 耐心表达，提示学生检查思路 |
| 小白板 | whiteboard | 移动到白板附近并展示关键词或步骤 |
| 暂停 | paused | 停止语音和主要动作，保留低频微动作 |
| 完成 | completed | 总结、自然收尾并返回待机或默认站位 |
| 退场 | leaving | 在隐藏或课程结束时自然离开 |
| 异常 | error | 停止复杂动作并进入固定或静态降级模式 |

### 3.3 状态优先级

```text
异常 > 用户打断 > 页面销毁 > 倾听 > 思考 > 移动 > 讲课 > 强调 > 鼓励/引导 > 待机
```

业务规则：

- 用户打断时立即取消当前语音、口型、移动和动作；
- 移动和主要身体动作不能同时由两个任务控制；
- 强调是讲课的临时子状态，结束后返回讲课；
- 鼓励和引导结束后根据业务流程进入倾听、讲课或待机；
- 思考状态必须设置最大持续时间，超时后进入普通等待或异常提示；
- 页面切换和组件销毁优先于所有教学动作；
- 非必要自主移动不得打断学生输入和阅读。

### 3.4 教师表现引擎

教师表现引擎接收业务内容和场景信息，输出统一表现脚本。

输入内容包括：

- 来源模块；
- 场景类型；
- 教学文本；
- 内容语义标签；
- 当前锚点；
- 学生回答状态；
- M6 提供的讲解偏好；
- 用户选择的教师风格和表现强度；
- 当前设备性能等级。

输出内容包括：

- 目标站位或锚点；
- 移动方式和速度；
- 身体朝向和视线目标；
- 表情和动作；
- TTS 音色、语速、音量和重读标签；
- 讲解前后停顿；
- 是否等待学生输入；
- 是否返回默认站位。

表现脚本示例：

```json
{
  "scene": "lecture",
  "contentType": "key_concept",
  "targetAnchor": "formula-domain-01",
  "targetPosition": "left_of_anchor",
  "movement": "walk",
  "expression": "serious",
  "motion": "point_right",
  "speechStyle": "slow_emphasis",
  "text": "这里需要特别注意函数的定义域。",
  "pauseBeforeMs": 300,
  "pauseAfterMs": 1000,
  "waitForStudent": false,
  "returnPosition": "right_stage"
}
```

### 3.5 教师风格

系统面向大学生提供三种教师风格：

| 教师风格 | 表达特点 | 动作特点 | 推荐场景 |
|---|---|---|---|
| 学术导师型 | 严谨、完整、强调概念与推导 | 动作克制，重点指向明确 | 理论课程、论文阅读、公式推导 |
| 友好学长学姐型 | 自然、平等、适度鼓励和轻松幽默 | 表情较自然，互动反馈适中 | 日常学习、答疑和复习 |
| 高效教练型 | 简洁、直接、突出重点和方法 | 动作较少，节奏较快 | 考前复习、做题和知识总结 |

不同风格可以使用同一模型，通过措辞、音色、语速、动作频率和表情幅度体现，不要求必须制作三套人物模型。

### 3.6 表现强度

| 模式 | 动作频率 | 自主移动 | 表情变化 | 适用场景 |
|---|---|---|---|---|
| 活跃 | 较高 | 允许低频移动和更多道具互动 | 较丰富 | 互动课堂、轻松学习 |
| 标准 | 适中 | 仅在教学需要时移动 | 自然 | 默认讲课、讲题和问答 |
| 专注 | 较低 | 禁止非必要自主移动 | 简洁 | 阅读、考试、代码编写和深度学习 |

系统默认使用标准模式。进入限时练习、考试或长文本阅读时，可提示用户切换专注模式。

### 3.7 虚拟教学舞台

教学页面定义以下基础站位：

| 站位 | 用途 |
|---|---|
| 左侧站位 | 主要内容位于右侧时使用 |
| 右侧站位 | 主要内容位于左侧时使用 |
| 中央站位 | 欢迎、总结和直接交流 |
| 白板站位 | 展示公式、关键词和推导步骤 |
| 题目站位 | 题目分析、选项提示和错误讲解 |
| 代码站位 | 靠近代码区域并指出错误行 |
| 对话站位 | 倾听学生提问和实时问答 |
| 默认待机站位 | 无明确任务时使用 |

人物优先在预设站位之间移动。需要靠近具体内容时，由教学锚点系统生成锚点附近的安全候选位置。

### 3.8 模块关系

```text
M2 AI 讲题 ───────┐
M4 即时讲课 ───────┼──> 教师表现引擎 ──> 状态机 ──> Live2D 渲染与空间控制
M6 上下文记忆 ─────┤          │                 ├──> 表情、动作与视线
M7 实时问答 ───────┘          │                 ├──> TTS 与口型同步
                               └──> 用户设置 ─────┴──> 性能监控与降级
```

### 3.9 核心业务流程

1. 学生进入讲课、讲题或问答页面；
2. M8 加载用户设置、模型配置和资源版本；
3. 模型加载完成后进入待机或入场状态；
4. M2、M4 或 M7 提交教学事件和表现上下文；
5. M8 从 M6 获取当前场景所需的最小学习偏好摘要；
6. 教师表现引擎生成表现脚本；
7. 空间控制器确定目标站位或锚点附近候选位置；
8. 遮挡检测器排除覆盖主要内容的候选位置；
9. 教师转身并播放移动动作，同时更新模型坐标；
10. 到达后调整视线、表情和身体朝向；
11. TTS 控制器生成或读取缓存音频；
12. 音频播放时启动口型同步，并按脚本播放动作；
13. 讲解结束后等待学生、返回原位或继续下一教学任务；
14. 用户打断时立即取消任务并进入倾听状态；
15. 系统记录加载、帧率、动作、语音和降级指标。

## 四、功能需求

### 4.1 FR-01 Live2D 模型加载与资源管理

#### 4.1.1 功能说明

系统应支持加载已审核和配置的 Live2D 教师模型，在教学页面透明渲染层中显示。

#### 4.1.2 功能要求

- 支持至少两种虚拟教师形象；
- 支持模型切换、缩放、位置和朝向配置；
- 支持模型、纹理、动作和表情资源异步加载；
- 支持资源版本和浏览器缓存；
- 支持加载进度展示；
- 模型加载不得阻塞题目、课件和文字内容显示；
- 模型加载失败时自动切换静态形象；
- 页面离开时释放 WebGL、音频和定时器资源；
- 模型资源必须具有明确授权，未经授权的网络模型不得进入项目。

#### 4.1.3 模型制作流程

允许采用以下模型来源：

1. 团队原创或委托绘制的分层原画；
2. 具有项目展示和修改授权的合法素材；
3. 用于技术验证的官方示例模型。

模型制作流程：

```text
角色设定 → 原画或授权素材 → Photoshop 修正与分层 → 隐藏区域补全
→ Live2D 参数绑定 → 表情与动作制作 → 性能检查 → 导出并接入网页
```

### 4.2 FR-02 微动作与自然待机

系统应支持：

- 自然呼吸；
- 随机眨眼；
- 轻微头部和身体重心变化；
- 低频视线移动；
- 说话前轻微准备动作；
- 倾听时轻微点头；
- 长时间待机时随机播放小幅动作。

业务规则：

- 微动作必须设置随机间隔，避免固定周期重复；
- 页面不可见时暂停或降低更新频率；
- 专注模式下只保留呼吸、眨眼和必要口型；
- 同一明显微动作不得短时间重复；
- 用户可关闭非必要动画。

### 4.3 FR-03 表情控制

基础表情至少包括：默认、微笑、认真、思考、疑问、耐心、鼓励和轻微惊喜。

业务规则：

- 表情切换应使用平滑过渡；
- 表情持续时间可配置；
- 表情结束后返回当前状态对应的基础表情；
- 学生答错时不得使用失望、嘲讽或负面表情；
- 不存在的表情自动回退为默认表情。

### 4.4 FR-04 动作与手势控制

基础动作至少包括：

- 入场和退场；
- 点头；
- 左右指向；
- 强调重点；
- 轻度认可；
- 思考；
- 倾听；
- 白板书写；
- 到达站稳；
- 行走或移动循环；
- 课程结束。

动作规则：

- 同一时刻只允许一个主要身体动作；
- 高优先级动作可中断低优先级动作；
- 同一明显动作设置冷却时间；
- 动作结束后返回当前状态；
- 动作不得遮挡内容或操作区域；
- 不同模型通过逻辑动作名称映射实际 motion 文件。

### 4.5 FR-05 教学语义驱动表现

系统应支持以下语义标签：

| 内容类型 | 推荐表现 |
|---|---|
| 欢迎 | 微笑、入场、自然问候 |
| 普通讲解 | 标准语速、普通讲课动作 |
| 重点概念 | 认真表情、降低语速、指向或强调 |
| 公式推导 | 移动到白板、逐步展示、增加停顿 |
| 代码错误 | 移动到代码锚点、指出错误行 |
| 提问 | 疑问表情、停止语音、等待学生 |
| 正确反馈 | 微笑或轻微点头，给出具体认可 |
| 接近正确 | 鼓励继续思考，不直接公布答案 |
| 错误反馈 | 耐心表情，提示检查关键步骤 |
| 总结 | 回到中央或默认站位，语速稍慢 |

系统应允许 M2、M4 和 M7 显式指定标签，也可以在安全规则范围内根据结构化教学内容选择预设标签。

### 4.6 FR-06 大学生教师风格

学生可选择学术导师型、友好学长学姐型或高效教练型。

要求：

- 风格影响措辞、语速、停顿、动作频率和表情幅度；
- 不改变知识结论和核心教学内容；
- 不使用低龄化称呼；
- 不频繁使用“你太棒了”等夸张表达；
- 用户当前明确要求优先于历史风格偏好；
- M6 可提供用户长期偏好，但用户可随时修改。

### 4.7 FR-07 趣味互动

系统可提供以下轻量趣味功能：

- 与课程内容相关的低频幽默表达；
- 完成章节后的简短庆祝动作；
- 公式卡、代码提示卡、成就卡和小白板；
- 快速判断、选项预测和十秒观察等互动挑战；
- AI 思考期间的自然等待动作；
- 根据本地时间进行简短问候；
- 对学生进步给出具体反馈。

限制：

- 幽默不得讽刺学生、成绩、能力或错误；
- 严肃内容和专注模式下默认减少幽默；
- 所有趣味功能可关闭；
- 不允许大模型自由生成未经约束的敏感玩笑；
- 同类趣味动作必须设置冷却时间。

### 4.8 FR-08 TTS 语音合成

系统应支持：

- 至少两种音色；
- 语速和音量设置；
- 长文本分段生成；
- 语音播放、暂停、继续和停止；
- 数字、英文、代码和常见公式的朗读预处理；
- 重点词重读、句间停顿和提问停顿；
- TTS 失败重试；
- TTS 异常时保留字幕和文字内容。

文本分段优先按照句号、问号、感叹号、段落和教学步骤处理。不得朗读 Markdown 标记、无意义符号和内部字段名称。

### 4.9 FR-09 音频缓存

缓存键建议：

```text
tts:{textHash}:{voiceId}:{speed}:{style}:{audioFormat}:{ttsVersion}
```

业务规则：

- 相同文本、音色、语速和风格优先使用缓存；
- 任一关键参数变化时重新生成；
- 常用课程音频可设置较长有效期；
- 私密问答音频采用短期或会话级缓存；
- 缓存失效不得影响正在播放的音频；
- 缓存异常时直接回源 TTS 服务。

### 4.10 FR-10 口型同步

MVP 使用音频振幅驱动：

1. 通过 Web Audio API 获取音频时域或频率数据；
2. 对音量进行归一化和平滑；
3. 映射至 Live2D 嘴部开合参数；
4. 暂停、停止和打断时将嘴部参数归零。

后续升级为音素或 viseme 时间轴驱动，实现不同发音口型。

要求：

- 音频开始后口型及时启动；
- 暂停和打断时口型在 500ms 内停止；
- 音频结束后嘴部恢复默认；
- 分析失败时使用轻量随机口型或关闭口型；
- 口型控制不得覆盖模型必要的表情参数。

### 4.11 FR-11 教学舞台与预设站位

系统应支持至少以下站位：左侧、右侧、中央、白板、题目、代码和对话站位。

每个站位至少包含：

- 站位编号；
- 视口比例坐标；
- 模型缩放比例；
- 默认朝向；
- 适用场景；
- 安全边距；
- 是否允许用户拖动覆盖。

窗口尺寸变化后，应根据比例重新计算站位。

### 4.12 FR-12 教学锚点

课件、题目、公式、图表和代码可注册为教学锚点。

锚点至少包含：

| 字段 | 说明 |
|---|---|
| anchorId | 锚点唯一编号 |
| anchorType | formula、chart、question、code、text 等类型 |
| element | 对应 DOM 元素 |
| preferredSide | 推荐教师站在左侧或右侧 |
| pointDirection | 推荐指向方向 |
| safeMargin | 教师与内容的最小距离 |
| allowApproach | 是否允许教师靠近 |

前端可通过 `data-teacher-anchor` 属性注册锚点。

### 4.13 FR-13 空间移动

一次完整移动应包含：

1. 取消冲突动作；
2. 看向移动方向；
3. 播放转身动作；
4. 播放行走或移动动作；
5. 使用缓动函数改变模型坐标和缩放；
6. 到达目标后减速；
7. 调整朝向和视线；
8. 播放到达、指向或讲解动作。

业务规则：

- 正常场景不得直接瞬移；
- 页面重排、模型异常或低性能降级时可使用淡入淡出；
- 移动中收到打断必须立即停止；
- 同一时间只执行一个主要移动任务；
- 人物不得超出可视区域；
- 专注模式禁止非必要自主移动。

### 4.14 FR-14 遮挡检测与候选站位评分

系统应收集以下障碍区域：

- 当前讲解内容；
- 输入框和编辑器；
- 题目选项；
- 播放和操作按钮；
- 字幕区域；
- 弹窗和通知；
- 用户指定的不可遮挡区域。

系统生成目标左侧、右侧、左下、右下和默认站位等候选位置，并按以下因素评分：

- 是否与障碍区域相交；
- 与目标锚点距离；
- 是否超出视口；
- 是否需要跨越主要内容；
- 与当前站位移动距离；
- 用户设置的偏好位置。

第一阶段不要求实现复杂自由寻路。若所有候选位置均不可用，教师保持安全站位，仅通过远距离视线和手势讲解。

### 4.15 FR-15 视线与朝向

系统应根据目标内容位置控制：

- 眼球水平和垂直参数；
- 头部角度；
- 身体轻微朝向；
- 模型左右翻转或专用转向动作。

视线变化应采用平滑插值，避免瞬间跳转。人物指向目标时，眼睛、头部和手势方向应基本一致。

### 4.16 FR-16 用户拖动与控制

学生应能够：

- 拖动教师位置；
- 调整教师大小；
- 锁定教师位置；
- 吸附到页面边缘；
- 恢复推荐站位；
- 隐藏教师；
- 关闭自主移动；
- 关闭语音、口型、幽默和非必要动作；
- 切换教师风格和表现强度；
- 召唤教师讲解选中的锚点内容。

用户手动位置在当前场景内优先于系统默认位置。切换新场景后可重新应用推荐站位，但不得无提示覆盖用户明确锁定的位置。

### 4.17 FR-17 小白板与轻量道具

系统应支持在教师附近显示轻量白板或提示卡，用于：

- 显示关键词；
- 展示公式推导关键步骤；
- 标出代码错误行；
- 圈出题目易错位置；
- 展示当前问题；
- 显示阶段总结或成就反馈。

道具内容必须同步提供普通 DOM 文本，不能只作为 Canvas 图像存在。

### 4.18 FR-18 M2 AI 讲题联动

| M2 场景 | M8 表现 |
|---|---|
| 正在识别或分析题目 | 思考状态，限制最大等待时长 |
| 开始讲解 | 移动至题目站位并进入讲课 |
| 展示关键步骤 | 靠近对应锚点，认真表情和强调动作 |
| 学生表示听懂 | 微笑或点头，给出具体反馈 |
| 学生表示没听懂 | 耐心表情，切换讲解方式 |
| 学生回答正确 | 轻度认可，不夸张庆祝 |
| 学生回答错误 | 引导检查错误位置，不表现失望 |
| 讲题结束 | 总结并返回默认站位 |

### 4.19 FR-19 M4 即时讲课联动

M4 应传递课程页码、教学阶段、重点标签、互动状态和内容锚点。

| M4 场景 | M8 表现 |
|---|---|
| 课程开始 | 自然入场和问候 |
| 正常讲解 | 讲课状态和口型同步 |
| 页面翻页 | 根据新布局选择安全站位 |
| 出现重点 | 移动到锚点附近并强调 |
| 公式推导 | 移动到白板，逐步展示 |
| 互动提问 | 回到中央或对话站位并等待 |
| 用户暂停 | 停止语音和主要动作 |
| 课程总结 | 回到中央进行收束 |

### 4.20 FR-20 M6 上下文记忆联动

M8 仅使用 M6 提供的场景化最小摘要，例如：

- 偏好图示、文字或语音；
- 喜欢先看例子还是先看定义；
- 学习节奏偏细致、适中或偏快速；
- 当前知识点的近期困惑；
- 用户主动选择的教师风格。

M8 可据此调整：

- 语速和停顿；
- 是否优先使用小白板或图示；
- 动作频率；
- 是否回顾上次卡住的位置；
- 教师风格默认值。

用户当前指令和设置始终优先于历史偏好。M8 不直接读取 M6 的完整底层事件。

### 4.21 FR-21 M7 实时问答联动

M7 状态流程：

```text
listening → thinking → speaking → completed
```

要求：

- 学生输入或说话时进入倾听状态；
- AI 生成回答时进入思考状态；
- 首段语音可播放后进入回答状态；
- 用户打断后 500ms 内停止语音、口型、动作和移动；
- 网络恢复后根据 M7 状态重新同步；
- AI 异常时不得无限保持思考状态。

### 4.22 FR-22 管理端资源配置

管理员应能够：

- 查看模型、音色、表情和动作列表；
- 配置逻辑动作与模型实际资源的映射；
- 设置默认模型、音色和教师风格；
- 启用、停用和回滚模型版本；
- 配置模型适用终端和性能等级；
- 查看加载失败、动作缺失和平均帧率；
- 管理趣味表达模板和禁用规则。

### 4.23 FR-23 异常与降级处理

| 异常场景 | 系统处理 |
|---|---|
| 模型加载失败 | 重试一次，随后使用静态教师图片 |
| 动作不存在 | 使用默认动作并记录警告 |
| 表情不存在 | 使用默认表情 |
| TTS 不可用 | 保留字幕和文字，提示语音暂不可用 |
| 音频加载失败 | 重新获取一次，失败后跳过语音 |
| 口型分析失败 | 使用轻量口型或关闭口型 |
| 帧率持续过低 | 逐级关闭物理效果、微动作和移动 |
| WebGL 不可用 | 静态形象或普通图片模式 |
| 锚点不存在 | 使用默认安全站位讲解 |
| 无可用候选站位 | 保持当前位置并远距离指向 |
| 用户打断 | 立即取消当前任务并进入倾听 |
| 页面销毁 | 停止音频并释放全部资源 |

## 五、接口需求

### 5.1 对外接口一览

| 方法 | 接口 | 用途 |
|---|---|---|
| GET | `/api/virtual-teacher/models` | 获取可用教师模型 |
| GET | `/api/virtual-teacher/models/{modelId}` | 获取模型详细配置 |
| GET | `/api/virtual-teacher/voices` | 获取可用音色 |
| GET | `/api/virtual-teacher/styles` | 获取教师风格和表现强度 |
| GET | `/api/virtual-teacher/settings` | 获取用户设置 |
| PUT | `/api/virtual-teacher/settings` | 修改用户设置 |
| POST | `/api/virtual-teacher/tts` | 生成或读取缓存语音 |
| POST | `/api/virtual-teacher/events` | 记录关键交互事件 |
| POST | `/api/virtual-teacher/metrics` | 上报性能与异常指标 |

### 5.2 前端控制接口

```typescript
interface VirtualTeacherController {
  initialize(config: TeacherConfig): Promise<void>;
  setState(state: TeacherState): void;
  execute(script: TeachingPerformanceScript): Promise<void>;
  moveToStation(stationId: string): Promise<void>;
  moveToAnchor(anchorId: string): Promise<void>;
  lookAtAnchor(anchorId: string): void;
  setExpression(expression: string): void;
  playMotion(motion: string, priority?: number): Promise<void>;
  speak(request: SpeechRequest): Promise<void>;
  pauseSpeech(): void;
  resumeSpeech(): void;
  interrupt(): void;
  setVisible(visible: boolean): void;
  destroy(): void;
```

## 七、技术方案

### 7.1 推荐技术栈

| 层级 | 技术 | 主要职责 |
|---|---|---|
| 前端框架 | React + TypeScript | 页面组件、状态管理和模块联动 |
| Live2D | Cubism SDK for Web 或兼容运行库 | 模型加载、参数、动作和表情控制 |
| 渲染层 | Canvas + WebGL，可选 PixiJS | 透明全屏渲染、坐标和图层管理 |
| 动画调度 | requestAnimationFrame | 人物位置、缩放和视线的逐帧更新 |
| DOM 定位 | getBoundingClientRect、ResizeObserver | 获取锚点和障碍区域位置 |
| 拖动交互 | Pointer Events | 鼠标和触摸拖动人物 |
| 音频层 | HTMLAudioElement + Web Audio API | 播放、音量分析和口型驱动 |
| 后端 | Spring Boot | 模型配置、用户设置、TTS 调用和日志 |
| 数据层 | MySQL + Redis | 配置、缓存、事件和指标 |
| 实时交互 | 组件事件、SSE 或 WebSocket | M7 流式状态和实时打断 |

### 7.2 前端分层

```text
VirtualTeacherSystem
├── Live2DRenderer
├── ModelAdapter
├── TeacherStateMachine
├── PerformanceScriptEngine
├── SpatialController
├── PathController
├── AnchorRegistry
├── CollisionAvoidanceController
├── GazeController
├── ExpressionController
├── MotionController
├── SpeechController
├── LipSyncController
├── PropController
├── UserInteractionController
└── PerformanceMonitor
```

职责说明：

- `Live2DRenderer`：初始化 WebGL 和绘制模型；
- `ModelAdapter`：屏蔽不同模型的参数和动作差异；
- `TeacherStateMachine`：管理状态和优先级；
- `PerformanceScriptEngine`：将业务事件转换为表现脚本；
- `SpatialController`：控制人物坐标、缩放和朝向；
- `PathController`：执行转身、行走、到达和取消；
- `AnchorRegistry`：注册和更新教学锚点；
- `CollisionAvoidanceController`：检测遮挡并选择安全位置；
- `GazeController`：控制眼球、头部和身体视线；
- `SpeechController`：TTS、播放队列和打断；
- `LipSyncController`：音频到嘴部参数的映射；
- `PropController`：小白板和提示卡；
- `PerformanceMonitor`：帧率、加载耗时和自动降级。

### 7.3 页面渲染层

教学页面上方设置透明全屏 Canvas：

```css
.teacher-layer {
  position: fixed;
  inset: 0;
  width: 100vw;
  height: 100vh;
  pointer-events: none;
  z-index: 100;
}
```

人物拖动时，可使用独立透明交互区域或临时开启局部 Pointer Events。Canvas 不应拦截课件、输入框和按钮的正常操作。

### 7.4 模型适配器

业务逻辑仅使用逻辑名称，例如：

```text
mouthOpen、eyeX、eyeY、headX、bodyX
walk、pointLeft、pointRight、think、listen、arrive
```

不同模型通过配置映射至实际参数 ID 和 motion 文件，避免切换模型后重写业务代码。

### 7.5 人物移动实现

人物移动由两部分同时完成：

1. Live2D 内部播放走路、转身或身体摆动动作；
2. 外部使用 `requestAnimationFrame` 平滑改变整个模型的 x、y 和 scale。

核心数据：

```typescript
interface TeacherTransform {
  x: number;
  y: number;
  scale: number;
  facing: "left" | "right";
}
```

移动使用缓动函数，不直接改变终点坐标。窗口变化时按视口比例重新计算站位。

### 7.6 教学锚点定位

教学元素使用：

```html
<div data-teacher-anchor="formula-domain-01">...</div>
```

前端通过 `getBoundingClientRect()` 获取其视口坐标，并用 `ResizeObserver` 监听课程布局、侧边栏和窗口尺寸变化。

### 7.7 遮挡检测

第一阶段使用矩形相交检测：

- 将教师估算为一个屏幕矩形；
- 收集所有 `data-teacher-obstacle` 区域；
- 生成锚点左、右、左下、右下和默认站位；
- 排除相交和超出视口的位置；
- 按距离、移动成本和用户偏好评分；
- 选择得分最高的位置。

不在 MVP 中实现复杂游戏寻路，避免技术成本过高。

### 7.8 移动任务和取消机制

每次教学行为应通过 `AbortController` 或统一取消令牌执行：

```text
确认目标 → 转身 → 行走 → 到达 → 看向目标 → 动作 → 语音 → 返回
```

用户打断时统一执行：

- 取消当前异步任务；
- 停止音频；
- 停止移动帧；
- 停止主要动作；
- 嘴部参数归零；
- 进入倾听状态。

### 7.9 视线实现

根据人物位置和目标中心计算归一化方向，将结果平滑映射至眼球、头部和身体参数。视线参数使用插值更新，不允许瞬间跳转。

### 7.10 口型实现

MVP：

- 创建 `AudioContext` 和 `AnalyserNode`；
- 获取当前音频强度；
- 平滑音量数据；
- 映射至嘴部开合参数；
- 停止时归零。

优化版本：TTS 返回音素或 viseme 时间戳，根据发音类别切换嘴型。

### 7.11 Live2D 美术与绑定实施

项目不要求模块负责人必须从零绘制原画。建议采用：

- 委托或团队成员完成基础原画；
- 模块负责人使用 Photoshop 完成修正、拆分和隐藏区域补图；
- 模块负责人完成 Live2D 参数绑定、表情、动作和性能优化；
- 第一阶段可使用合法示例模型验证程序功能；
- 第二阶段替换为正式大学生教师形象。

为支持空间移动，正式模型建议至少拆分：头部、前后发、五官、脖子、躯干、左右上臂、左右前臂、左右手、上下身和必要道具。完整双腿行走可列为优化项；MVP 可采用大半身移动、身体摆动和短距离过渡。

### 7.12 性能降级

降级等级：

| 等级 | 表现 |
|---|---|
| L0 完整模式 | 物理效果、微动作、移动、口型和道具全部开启 |
| L1 轻量模式 | 降低头发和衣服物理精度，减少微动作 |
| L2 基础模式 | 关闭非必要道具和自主移动，只保留口型与必要动作 |
| L3 固定模式 | Live2D 固定在安全位置，不执行长距离移动 |
| L4 静态模式 | 使用静态教师图片和字幕，可保留普通语音 |

当平均帧率连续 5 秒低于阈值时自动降级。性能恢复后不应频繁来回切换，可在下一课程或用户确认后升级。

## 八、非功能需求

### 8.1 性能

| 指标 | 要求 |
|---|---|
| 普通设备平均帧率 | 不低于 30fps |
| 推荐设备平均帧率 | 目标达到 50fps 以上 |
| 缓存命中模型首帧 | 不超过 2 秒 |
| 未缓存模型首帧 | 目标不超过 5 秒 |
| 状态切换 P95 | 不超过 200ms |
| 普通动作触发 P95 | 不超过 300ms |
| 用户打断停止语音 | 不超过 500ms |
| 用户打断停止口型和移动 | 不超过 500ms |
| TTS 缓存命中 P95 | 不超过 300ms |
| TTS 未命中首段可播放 | 目标不超过 3 秒 |
| 锚点定位 P95 | 不超过 100ms |
| 候选站位计算 P95 | 不超过 100ms |
| 连续运行 | 30 分钟无明显持续内存增长 |

### 8.2 稳定性

- 模型异常不得影响文字、题目和课件；
- TTS 异常不得影响文字回答；
- 单个动作缺失不得导致组件崩溃；
- 页面退出后不得继续播放音频；
- 不得重复播放同一语音；
- 网络恢复后能够重新同步状态；
- 状态异常时回到待机或固定模式。

### 8.3 兼容性

优先支持最新两个主要版本的 Chrome、Edge、Safari 和 Firefox。浏览器缺少 WebGL 或 Web Audio API 时，使用静态形象和普通音频模式。

### 8.4 可访问性

- 所有语音必须同步提供字幕；
- 动画不能作为知识信息的唯一载体；
- 用户可以关闭动画、移动和自动播放；
- 支持键盘操作和清晰焦点；
- 不使用高频闪烁；
- 支持减少动态效果；
- 小白板内容同时提供可访问的 DOM 文本。

### 8.5 安全与隐私

- TTS 请求不得包含令牌、密码和无关隐私信息；
- 私密问答音频采用短期缓存；
- 模型和音频资源实施访问控制；
- 模型配置不得执行未知脚本；
- 日志不记录完整私密文本；
- 第三方 TTS 调用符合平台隐私策略；
- M6 仅提供最小场景摘要，M8 不读取完整画像。

### 8.6 内容与人格安全

- 教师不得讽刺、羞辱或贬低学生；
- 不以单次错误推断学生能力；
- 幽默模板必须可审核和禁用；
- 不使用不合时宜、歧视性或攻击性表达；
- 严肃课程和专注模式减少娱乐表现；
- 教师风格不得改变事实、公式和知识结论。

### 8.7 可维护性

- 状态、动作、表情、风格和事件使用统一枚举；
- 模型差异通过配置适配；
- 接口和表现脚本带版本号；
- 核心控制器具有单元测试；
- 模型资源支持版本回滚；
- M2、M4、M6、M7 与 M8 建立契约测试；
- 核心类和公共接口编写中文注释或文档。

| AC-25 | 资源释放 | 页面退出后音频停止，模型和监听器释放 |
| AC-26 | 字幕 | 所有语音内容同步显示文字 |

### 9.2 自然度验收

1. 表情切换不存在明显瞬间跳变；
2. 行走动作与模型位移方向基本一致；
3. 人物移动前有转身或朝向变化；
4. 到达后先站稳再开始主要讲解动作；
5. 同一明显动作不会短时间连续重复；
6. 学生提问时人物能够及时停止讲解并倾听；
7. 思考状态不会无限持续；
8. 人物移动不会持续遮挡主要内容；
9. 专注模式不会出现非必要自主移动；
10. 教师反馈语言不存在低龄化、讽刺或过度夸张。

### 9.3 测试要求

- 模型加载测试覆盖首次、缓存、资源缺失和版本更新；
- 状态机测试覆盖正常、非法转换和高优先级打断；
- 移动测试覆盖取消、窗口缩放和锚点消失；
- 遮挡测试覆盖输入框、弹窗和课件重点；
- TTS 测试覆盖长文本、代码、公式和超时；
- 缓存测试覆盖命中、未命中、过期和异常；
- 口型测试覆盖播放、暂停、继续、结束和打断；
- M2、M4、M6、M7 联动执行契约测试；
- 低配置设备执行性能和降级测试；
- 主流浏览器执行兼容性测试；
- 字幕、键盘和减少动态效果执行可访问性测试。

## 十、实施优先级

本方案将需求分为三类：

- **P0 必须完成**：形成可运行、可演示、可验收的 M8 基础闭环；
- **P1 重点创新**：体现本模块区别于普通虚拟头像的核心特色；
- **P2 后续优化**：在时间和资源允许时继续提升自然度与完整度。

### 10.1 第一阶段：MVP（P0 必须完成）

1. 接入一个合法 Live2D 示例或正式模型；
2. 建立透明全屏 Canvas；
3. 实现待机、倾听、思考、移动和讲课状态；
4. 实现呼吸、眨眼和基础视线；
5. 实现左、中、右三个站位；
6. 实现模型坐标平滑移动和基础行走动作；
7. 接入一种 TTS 音色；
8. 实现语音播放、暂停、停止和打断；
9. 使用音频振幅实现基础口型；
10. 完成 M4 一个页面和一个公式锚点的演示；
11. 模型异常时使用静态图片；
12. 完成基础帧率监控。

### 10.2 第二阶段：功能完善（P1 重点创新）

1. 增加第二种教师形象和第二种音色；
2. 增加白板、题目、代码和对话站位；
3. 建立教学锚点注册和布局更新；
4. 实现候选站位评分和遮挡检测；
5. 完成 M2 和 M7 联动；
6. 接入 M6 场景化偏好摘要；
7. 增加三种教师风格和三档表现强度；
8. 增加音频缓存；
9. 增加小白板和关键词卡；
10. 增加用户拖动、锁定和关闭自主移动；
11. 完成完整、轻量、固定和静态降级。

### 10.3 第三阶段：体验优化（P2 后续优化）

1. 优化动作与语义匹配；
2. 增加更自然的转身、到达和收尾；
3. 引入音素时间轴口型；
4. 增加课程相关的道具和低频趣味互动；
5. 完善大学生教师表达模板；
6. 支持更复杂的页面锚点和滚动跟随；
7. 增加模型和动作效果评估；
8. 根据学生反馈优化动作频率、语速和教师风格。

### 10.4 最终交付判定

项目达到以下条件时，可认定 M8 最终方案完成基础交付：

1. 至少一个正式或合法示例 Live2D 模型可稳定渲染；
2. 待机、倾听、思考、移动、讲课和打断状态形成闭环；
3. 教师可在至少三个预设站位之间平滑移动；
4. 至少完成一个公式、题目或代码教学锚点的靠近、注视、指向和讲解演示；
5. TTS、字幕、基础口型和 500ms 内打断可正常工作；
6. 模型异常时不影响文字、题目、课件和问答；
7. 普通测试设备平均帧率达到 30fps，低性能时可以降级；
8. 完成 M4 基础联动，并为 M2、M6、M7 提供统一接口或事件契约；
9. 学生可以隐藏、拖动或关闭自主移动；
10. 文档、接口、测试记录和演示材料完整。

## 十一、潜在风险与应对措施

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 原画制作能力不足 | 正式角色进度受阻 | MVP 使用合法示例模型，正式原画可委托，团队负责 PS 分层和绑定 |
| 模型资源过大 | 首帧慢、占用内存 | 压缩纹理、按需加载、CDN 和版本缓存 |
| 全身行走绑定复杂 | 延误 MVP | 第一阶段采用大半身移动和短距离过渡，完整腿部行走后置 |
| 低配置设备帧率低 | 动画卡顿 | 自动关闭物理效果、移动和道具，最终切换静态模式 |
| 动作过多 | 干扰学习 | 表现强度、冷却时间、专注模式和用户开关 |
| 动作与语义不匹配 | 降低可信度 | 使用结构化语义标签和预设动作，不让模型自由决定全部动作 |
| 人物遮挡内容 | 影响阅读与操作 | 锚点安全边距、障碍检测和备用站位 |
| 移动显得滑动 | 人物不自然 | 同步播放行走动作、转身、减速和到达动作 |
| 多模块状态冲突 | 动作混乱 | 统一状态机、任务队列、优先级和取消令牌 |
| TTS 延迟过高 | 讲解等待 | 长文本分段、缓存和首段优先播放 |
| 口型不同步 | 降低人感 | MVP 使用平滑振幅，后续升级音素时间轴 |
| 幽默不合适 | 伤害体验 | 使用审核模板、低频触发和关闭选项 |
| 浏览器能力不足 | 无法渲染 | 固定 Live2D、静态图片和文字降级 |
| 长时间运行内存增长 | 页面逐渐卡顿 | 组件销毁、对象复用、监听器清理和内存测试 |
| 模型授权不明确 | 法律与展示风险 | 使用官方示例或具有明确授权的原创、委托和购买素材 |

## 十三、结论

M8“虚拟 AI 教师形象（优化）”最终方案的核心不再只是让人物拥有表情和口型，而是建立一个能够理解教学场景并控制教师完整表现的系统。

本版本以大学生为唯一目标用户，强调专业、自然、平等和克制的互动方式。虚拟教师通过微动作、倾听、思考、视线、语音节奏和连续交流增强人感；通过教师风格、小白板、课程道具、互动挑战和低频幽默提升趣味性；通过教学舞台、预设站位、锚点、遮挡检测和空间移动突破固定侧边栏限制。

技术实现采用“Live2D 内部动作＋模型外部坐标移动”的组合方式，并使用“预设站位＋教学锚点＋候选位置评分”控制空间行为。该方案不依赖复杂三维技术和游戏级自由寻路，能够在现有 Web 技术范围内分阶段落地。

最终方案的开发策略是先完成稳定可演示的 MVP：一个模型、基础状态、三个站位、一种语音、音量口型和一个课件锚点；随后再逐步加入多模型、多音色、遮挡检测、教师风格、趣味互动和完整性能降级。通过这一设计，M8 将从“固定的动态头像”升级为能够走近内容、看向重点、自然讲解、认真倾听并参与学习过程的虚拟 AI 教师。
